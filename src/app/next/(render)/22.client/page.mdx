## use client

在 NextJS 中，默认情况下，组件都是服务端组件。我们可以使用 `use client` 表示一个客户端组件。

```ts page.tsx -c
// !mark
'use client'

import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  )
}
```

需要特别注意的是，他真实的含义是在**模块结构树**中，表达服务端与客户端组件的边界。因此，在这颗树的后续子模块中，无需使用 `use client` 来表示客户端组件，NextJS 会自动将其解析为在客户端运行的模块。

如下图所示

<TreeSVG />

因此，我们只需要在少数几个边界处使用 `use client` 即可，而无需在每个客户端模块都用它来标记。

理想情况下，我们应该尽可能的将客户端组件下放到更末尾的节点，以避免造成冗余的重复渲染。具体的运用我们在前面已经通过大量的实例来验证这个结论。


## 执行

客户端组件的执行包含两个部分。

首先，他**会在服务端**执行一次渲染用于初始化。这是许多初学者非常容易理解错误的地方。认为客户端组件仅在客户端执行。并不是这样。那么也就意味着，我们不能在客户端组件的创建函数内直接调用服务端无法使用的 API，例如

```ts page.tsx -c
'use client'

import { useState } from 'react'

export default function Counter() {
  const [count, setCount] = useState(0)

  // !diff -
  console.log(window.location.href)

  return (
    <div>
      <p>You clicked {count} times</p>
      <button onClick={() => setCount(count + 1)}>Click me</button>
    </div>
  )
}
```

此时会直接报错

```ts
Error: Switched to client rendering because the server rendering errored:

  window is not defined
    at page.tsx
    at ...
```

不过这样一来，也会有一个好处就是，我们不需要担心客户端组件无法享受到 SEO 的利好。不过这里也要小心的一点就是，客户端组件的初始化更新，由于在服务端渲染，因此无法从客户端拿到相关的数据和状态，因此我们在写代码时要有清晰的思路，明确哪些内容是在服务端渲染完成的，哪些内容是在客户端渲染出来的。和服务端组件一样，客户端组件在服务端渲染之后，也会包含最终的 HTML 结果与对应的 RSC Payload 数据。

然后，客户端组件会在客户端进行水合，并执行后续的更新渲染。首先，浏览器会渲染 HTML 片段用于快速展示预览，然后基于 RSC Payload 参与协调过程进行水合。
